# Benchmark Datasets

## Real-World Datasets

### BPI 2020 Travel Permits

| Property         | Value                                                              |
| ---------------- | ------------------------------------------------------------------ |
| **Source**       | BPI Challenge 2020 (4TU Research Data)                             |
| **URL**          | `https://data.4tu.nl/articles/dataset/BPI_Challenge_2020/12752820` |
| **Domain**       | Dutch government travel permit process                             |
| **Traces**       | 10,500                                                             |
| **Events**       | ~141,000                                                           |
| **Activities**   | ~20                                                                |
| **Format**       | XES 2.0 (XML)                                                      |
| **Activity key** | `concept:name`                                                     |
| **Case ID**      | Trace-level (standard XES trace identification)                    |

This dataset is used for real-world validation of all 21 algorithms. It contains noise, loops, and non-trivial control flow that stress-test discovery quality.

## Synthetic Datasets

### Generation Parameters

| Parameter       | Default  | Description                                |
| --------------- | -------- | ------------------------------------------ |
| `numCases`      | variable | Number of traces (100, 1000, 10000, 50000) |
| `numActivities` | 6        | Distinct activity names                    |
| `avgEvents`     | 20       | Average events per trace                   |
| `noiseFactor`   | 0.0      | Probability of inserting a noise event     |
| `seed`          | fixed    | Deterministic seed for reproducibility     |

### Standard Benchmark Sizes

| Size Label | Cases  | Approximate Events | Primary Use                     |
| ---------- | ------ | ------------------ | ------------------------------- |
| **100**    | 100    | ~2,000             | Startup latency, algorithm init |
| **1K**     | 1,000  | ~20,000            | Small-scale, interactive use    |
| **10K**    | 10,000 | ~200,000           | Standard benchmark (most cited) |
| **50K**    | 50,000 | ~1,000,000         | Large-scale, scaling validation |

### How Synthetic Logs Are Generated

Synthetic logs are generated by the built-in test fixture generator in `packages/testing/src/fixtures/`. The generator creates a structured process model with defined activity sequences and optional noise.

```typescript
// Fixture generation (simplified)
generateLog({
  numCases: 10000,
  numActivities: 6,
  avgEventsPerCase: 20,
  noiseFactor: 0.0,
  seed: 42,
});
```

All synthetic logs use the same seed for deterministic reproducibility across benchmark runs.

## Event Log Format: XES

### XES Structure (Relevant Fields)

```xml
<log>
  <trace>
    <string key="concept:name" value="case-1"/>
    <event>
      <string key="concept:name" value="Submit Application"/>
      <date key="time:timestamp" value="2020-01-01T09:00:00Z"/>
    </event>
    <event>
      <string key="concept:name" value="Approve"/>
      <date key="time:timestamp" value="2020-01-02T10:00:00Z"/>
    </event>
  </trace>
</log>
```

### Key XES Attributes

| Attribute           | Key                    | Type   | Description                       |
| ------------------- | ---------------------- | ------ | --------------------------------- |
| **Activity name**   | `concept:name`         | string | Activity performed in this event  |
| **Timestamp**       | `time:timestamp`       | date   | When the event occurred           |
| **Case identifier** | (trace-level)          | string | Groups events into traces         |
| **Lifecycle**       | `lifecycle:transition` | string | Optional: start/complete/withdraw |

## Obtaining Datasets

### BPI 2020

1. Visit `https://data.4tu.nl/articles/dataset/BPI_Challenge_2020/12752820`
2. Download the XES file (Travel Permits)
3. Place in your project directory or reference by path

### Synthetic Logs

No download needed. Generated at runtime by the test fixture generator:

```bash
# Benchmarks generate synthetic logs automatically
cd wasm4pm && npm run bench
```
