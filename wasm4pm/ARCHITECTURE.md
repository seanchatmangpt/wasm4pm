# Architecture of process_mining_wasm

This document explains the design, architecture, and internal structure of the process_mining_wasm module.

## Overview

process_mining_wasm is a WebAssembly (WASM) wrapper around the Rust `process_mining` library. It provides JavaScript/TypeScript bindings to process mining algorithms and data structures, enabling the use of high-performance process mining tools directly in web browsers and Node.js environments.

## Core Components

### 1. WASM Module (`process_mining_wasm.wasm`)

The compiled WebAssembly binary containing the Rust process_mining library. This file:
- Contains compiled Rust code
- Runs in the WASM runtime environment
- Provides high-performance computation
- Handles memory management for process mining data structures

**File size:** Typically 1-3 MB (optimized with LTO and size flags)

### 2. JavaScript Glue Code (`process_mining_wasm.js`)

Auto-generated by `wasm-bindgen`, this module:
- Exports the WASM module initialization
- Provides JavaScript function bindings
- Handles type marshalling between JS and WASM
- Manages memory for JavaScript objects
- Available in multiple target formats (bundler, nodejs, web)

### 3. TypeScript Definitions (`process_mining_wasm.d.ts`)

Type definitions generated by `wasm-bindgen`:
- Provides full IDE autocomplete
- Type checking for TypeScript projects
- Runtime safety for JavaScript code

## Memory Management

### Object Storage

The WASM module maintains an internal object store:

```
JavaScript                    WASM Runtime
┌──────────────────┐        ┌──────────────────┐
│ logHandle (str)  │───────→│ Object Store     │
│ modelHandle      │───────→│ ┌──────────────┐ │
│ statsHandle      │───────→│ │ EventLog     │ │
└──────────────────┘        │ │ PetriNet     │ │
                            │ │ Statistics   │ │
                            │ └──────────────┘ │
                            └──────────────────┘
```

**Key points:**
- Handles are opaque string identifiers
- Objects live in WASM linear memory
- Handles don't directly expose memory addresses
- Manual cleanup required via `delete_object()`

### Memory Lifecycle

```
1. Load/Create Object
   load_eventlog_from_xes() ──→ Returns handle
   
2. Use Object
   discover_dfg(handle) ──→ Returns new handle
   analyze_event_statistics(handle) ──→ Returns results
   
3. Export Data (Optional)
   export_eventlog_to_xes(handle) ──→ Returns JSON/XML string
   object_to_json(handle) ──→ Returns JSON string
   
4. Cleanup
   delete_object(handle) ──→ Frees memory
```

### Memory Optimization

The package uses several techniques to minimize memory usage:

1. **LTO (Link Time Optimization):** Enabled in release builds
2. **Size optimization:** Profile set to `opt-level = "z"` (optimize for size)
3. **Lazy initialization:** Some data structures are computed on demand
4. **Streaming:** Large exports can be streamed if needed

## Data Flow

### Loading Data

```
User provides XES content (string)
           ↓
load_eventlog_from_xes(content)
           ↓
wasm-bindgen marshals string to WASM
           ↓
Rust code parses XES XML
           ↓
EventLog struct created in WASM memory
           ↓
Handle (string ID) returned to JavaScript
```

### Processing Data

```
JavaScript code has handle
           ↓
Call algorithm: discover_dfg(handle)
           ↓
wasm-bindgen passes handle to WASM
           ↓
Rust code retrieves object from store
           ↓
Algorithm processes the data
           ↓
Result object created in WASM memory
           ↓
New handle returned to JavaScript
```

### Exporting Results

```
JavaScript has result handle
           ↓
Call export: object_to_json(handle)
           ↓
Rust code retrieves and serializes object
           ↓
JSON string created
           ↓
wasm-bindgen marshals string to JavaScript
           ↓
JSON string returned to user
           ↓
User parses with JSON.parse()
```

## API Binding Strategy

### Function Binding Patterns

All functions follow these patterns:

**Input-Output (IIO) Pattern:**
```rust
// Rust
#[wasm_bindgen]
pub fn discover_dfg(handle: String) -> String {
    // Get object from store
    // Process it
    // Store result
    // Return new handle or serialized data
}
```

**String-based Marshalling:**
- All complex data crosses the JS-WASM boundary as strings
- Reduces type translation complexity
- Simple: handles and JSON strings
- Flexible: any serializable type can be transmitted

**Error Handling:**
- Rust `Result` types are converted to JavaScript exceptions
- `?` operator in Rust becomes thrown errors in JavaScript
- Try-catch in JavaScript handles these errors

## Performance Characteristics

### Algorithm Complexity

Discovery algorithms have varying computational complexity:

- **Alpha++:** O(n²) to O(n³) depending on log size
- **DFG:** O(n) linear scan
- **OC-DFG:** O(n × m) where m = number of object types
- **DECLARE:** O(n × k²) where k = number of activities

### Execution Profile

```
Small logs (<1K events):     < 100ms
Medium logs (1K-100K):       100ms - 10s
Large logs (>100K events):   10s - minutes
```

### Memory Usage

- **Metadata:** ~100 bytes per event
- **Index structures:** ~200 bytes per activity
- **Results:** Varies by algorithm

## Design Decisions

### Why String-based Marshalling?

- **Simplicity:** No complex type mapping
- **Flexibility:** Any serializable type works
- **Debuggability:** Can inspect intermediate results
- **Stability:** Less dependent on wasm-bindgen internals

### Why Individual Handles?

- **Memory safety:** Objects can't be accessed after deletion
- **Explicit cleanup:** Users control resource lifetime
- **Thread safety:** Handles are opaque identifiers
- **Forwards compatibility:** Internal structure can change

### Why Multiple Build Targets?

- **Flexibility:** Different environments have different needs
- **Optimization:** Each target optimized for its use case
- **Compatibility:** Broader compatibility across JS ecosystems

## Future Improvements

1. **Streaming API:** For processing very large logs
2. **Worker threads:** Offload computation to workers
3. **Incremental discovery:** Stream results progressively
4. **Plugin system:** Allow custom algorithms
5. **Type conversions:** Direct TypeScript type support

## Security Considerations

1. **Input validation:** All inputs validated in Rust
2. **Memory safety:** Rust prevents buffer overflows
3. **No arbitrary code execution:** WASM sandbox limits scope
4. **Integrity:** Built from verified source code
